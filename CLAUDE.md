## Project Overview

ft_transcendence — a real-time multiplayer card game platform with blockchain integration (Avalanche C-Chain). React 18 SPA frontend, Express backend, MariaDB + Redis storage, all containerized with Docker Compose.

## Build & Run Commands

All commands run from the project root. The entire stack runs in Docker.

```bash
make dev_up       # Start development environment (client:5173, server:3000, mariadb, redis)
make dev_down     # Stop development containers
make dev_fclean   # Full dev cleanup (removes volumes, networks, images)
make re           # dev_fclean + dev_up (full rebuild)

make up           # Production build + start (nginx:8443 HTTPS)
make down         # Stop production
make fclean       # Full production cleanup
```

There are no test suites for client or server. The blockchain contracts have Hardhat tests:
```bash
cd server/src/blockchain/contracts/
npx hardhat test
```

No linting scripts are configured in package.json. ESLint/Prettier are installed as devDependencies in client but have no run scripts.

## Architecture

### Docker Services (docker/docker-compose.yml)

Profiles control which services start: `dev`, `prod`, `build`.

- **dev_client** — Vite dev server with hot reload, proxies `/api` to `dev_server:3000`
- **dev_server** — Express with nodemon, source-mounted for live reload
- **mariadb** — MariaDB 11, initialized from `mariadb/init.sql`, data persisted in `mariadb_data` volume
- **redis** — Redis 7, no persistence (in-memory only)
- **nginx** — Production only, HTTPS reverse proxy (port 8443), serves built React SPA
- All services communicate on the `appnet` bridge network

### Client (`client/`)

React 18 + Vite 5 + Tailwind CSS. Entry point: `src/app/main.jsx` → `src/app/app.jsx`.

**Import aliases** (defined in both `vite.config.js` and `jsconfig.json`):
`@app`, `@api`, `@socket`, `@components`, `@controllers`, `@form`, `@layout`, `@section`, `@ui`, `@contexts`, `@css`, `@hooks`, `@pages`, `@routes`, `@utils`

**State management**: React Context API — `UserContext` (auth/user state), `ToastContext` (notifications), `InputContext` (keyboard input).

**Pages**: LoginPage, SignupPage, HomePage (lobby), RoomPage (game room).

**Pattern**: Pages → Custom hooks (in `hooks/`) → Services (`services/api/` for REST, `services/socket/` for WebSocket).

### Server (`server/`)

Express 4 + Socket.IO + Sequelize ORM + ioredis. Entry point: `src/server.js`.

**Import aliases** (Node.js subpath imports in `package.json`):
`#src/*`, `#config/*`, `#db/*`, `#middleware/*`, `#plugins/*`, `#api/*`, `#socket/*`, `#utils/*`, `#blockchain/*`

**Key directories**:
- `services/api/` — REST routes (auth, user, games), initialized in `initApi.js`
- `services/socket/` — Socket.IO handlers (move, chat, heartbeat, disconnect), initialized in `initSocket.js`
- `middleware/` — JWT auth, socket auth, user validation, rate limiting
- `db/mysql/` — Sequelize connection + User model + CRUD
- `db/redis/` — Redis connection + wrapper functions for online users, rooms, signup queue
- `plugins/` — Business logic (auth cookies, email, chat, room management)
- `config/env.js` — All environment variable configuration

**Data split**: MariaDB stores persistent user data (accounts, stats). Redis stores real-time state (online users, game rooms, ready status, rate limits).

### Blockchain (`server/src/blockchain/`)

Solidity smart contracts on Avalanche Fuji testnet. Hardhat 3 for compilation/testing/deployment. ethers.js v6 for backend interaction. Purpose: immutable game result recording.

## Commit Convention

Conventional Commits format: `type(scope/domain): description`

Types: `feat`, `fix`, `refactor`, `chore`, `docs`

Scopes by area:
- Client: `client/config`, `client/auth`, `client/hook`, `client/component`, `client/page`
- Server: `server/api`, `server/socket`, `server/middleware`, `server/db`
- Infrastructure: `docker`, `redis`, `db`

## Environment

Configuration lives in `docker/.env` (auto-generated by Makefile). Key variables: `VITE_SERVER_IP` (auto-detected), `NODE_ENV`, MySQL credentials, JWT secrets, SMTP config. See `docker/.env.blockchain.example` for blockchain-specific vars.
